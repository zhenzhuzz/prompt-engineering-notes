## Claim

后端技术栈可按四层模型理解：Runtime（应用框架/RPC）→ Data（存储/缓存/消息）→ Infra（容器/编排/代理）→ Delivery（版本控制/CI-CD）。每层解决一类核心问题，先理解层的职责再学具体工具。

## Evidence

根据 EVI-20251222-0004#loc:term-list，将 16 个技术名词分为四层：

```
┌─────────────────────────────────────────────────────────────┐
│  Delivery Layer (交付层)                                     │
│  问题: 代码如何版本管理和自动部署？                            │
│  工具: Git, Jenkins                                          │
├─────────────────────────────────────────────────────────────┤
│  Infrastructure Layer (基础设施层)                           │
│  问题: 应用如何打包、运行、发现、负载均衡？                     │
│  工具: Docker, Kubernetes, Nginx, Nacos, Zookeeper          │
├─────────────────────────────────────────────────────────────┤
│  Data Layer (数据层)                                         │
│  问题: 数据如何存储、缓存、搜索、异步传递？                     │
│  工具: MySQL, PostgreSQL, Redis, Elasticsearch, Kafka, MQ   │
├─────────────────────────────────────────────────────────────┤
│  Runtime Layer (运行时层)                                    │
│  问题: 业务逻辑用什么框架写？服务间如何通信？                   │
│  工具: Spring Boot, gRPC, Dubbo                             │
└─────────────────────────────────────────────────────────────┘
```

## Transferable Rule

**先定位层，再选工具。** 当看到新技术名词时，先问"它属于哪一层？解决什么核心问题？"这比直接学工具用法更高效。层的职责相对稳定，工具会迭代。

**Open questions:**
- Kafka vs RabbitMQ 的选择边界需要进一步查证
- Nacos vs Zookeeper 的适用场景需要补充外部来源
