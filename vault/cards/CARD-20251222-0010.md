## Claim

Kubernetes 解决多容器编排、自动伸缩、服务发现问题，但对小规模/简单架构项目是 overkill。判断信号（启发式）：服务数 ≥ 5、需跨节点调度、需自动伸缩（HPA/VPA）时考虑 K8s；否则 Docker Compose 或 K3s 通常是更优选择。

## Evidence

根据 EVI-20251222-0005#loc:findings.kubernetes：

**K8s 解决的问题**：
- 多容器部署、调度、伸缩
- 服务发现与负载均衡
- 滚动更新与回滚
- 自愈（Pod 重启、节点迁移）

**Overkill 信号（何时 K8s 可能过度）**：
- 单机部署即可满足当前和可预见负载
- 服务数 < 5 且无复杂依赖编排
- 团队无 K8s 运维经验且学习时间有限
- 早期 MVP，运维成本敏感
- 无跨节点调度需求（单节点足够）

**Must-have 信号（何时应认真考虑 K8s）**：
- 需跨多节点自动调度工作负载
- 需基于 CPU/内存的自动伸缩（HPA/VPA）
- 多团队协作，需统一部署标准和 RBAC
- 需蓝绿/金丝雀部署等高级发布策略
- 已有微服务架构，服务数持续增长

**隐性成本**：
- 学习曲线陡峭（YAML、Pod/Node/Ingress 概念）
- 云端账单可能很高（auto-scaling 意外触发）
- 本地运行资源消耗大

## Transferable Rule

**K8s 边界决策清单（4 步）：**

```
Step 1: 检查调度需求
        □ 需要跨多台机器自动调度？ → 继续评估 K8s
        □ 单机足够？ → 停止，用 Docker Compose

Step 2: 检查伸缩需求
        □ 需要 HPA/VPA 自动伸缩？ → 倾向 K8s
        □ 手动扩容足够或负载稳定？ → Docker Compose/K3s 可能够用

Step 3: 评估运维能力
        □ 团队有 K8s 经验或愿意投入学习？ → K8s 可行
        □ 无经验且时间紧？ → 先用简单方案，延后迁移

Step 4: 考虑规模信号（启发式）
        □ 服务数 ≥ 5 且持续增长？ → K8s 收益更明显
        □ 服务数 < 5 且稳定？ → 简单方案通常够用
```

**停止条件**：若无跨节点调度需求，不要仅因"大家都用 K8s"而采用。

**轻量替代方案**：
- **Docker Compose** — 单机多容器、开发环境、中小规模生产
- **K3s** — 边缘/IoT/CI（轻量 K8s，降低运维开销）
- **Serverless** — 事件驱动、无需管基础设施

**快速判断**：问"我需要跨多台机器自动调度吗？"——不需要则先用 Docker Compose。
