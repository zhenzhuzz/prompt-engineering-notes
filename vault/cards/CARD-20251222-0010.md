## Claim

Kubernetes 解决多容器编排、自动伸缩、服务发现问题，但对小规模/简单架构项目是 overkill。判断边界：当"服务数量 < 5 且无精细伸缩需求"时，Docker Compose 或 K3s 是更优选择。

## Evidence

根据 EVI-20251222-0005#loc:findings.kubernetes：

**K8s 解决的问题**：
- 多容器部署、调度、伸缩
- 服务发现与负载均衡
- 滚动更新与回滚
- 自愈（Pod 重启、节点迁移）

**何时 overkill**：
- 小规模项目（单服务或少数服务）
- 简单 CRUD 应用不需要复杂编排
- 早期创业项目：运维成本 > 收益
- 团队缺乏 K8s 运维经验

**隐性成本**：
- 学习曲线陡峭（YAML、Pod/Node/Ingress 概念）
- 云端账单可能很高（auto-scaling 意外触发）
- 本地运行资源消耗大

## Transferable Rule

**K8s 边界决策清单：**

```
┌─────────────────────────────────────────────────────────────┐
│  需要 K8s 当：                                              │
│  □ 服务数量 ≥ 5，且需统一编排                               │
│  □ 需要精细伸缩策略（HPA、VPA）                             │
│  □ 多团队协作，需统一部署标准                               │
│  □ 团队有 K8s 运维经验                                      │
├─────────────────────────────────────────────────────────────┤
│  K8s 是 overkill 当：                                       │
│  □ 服务数量 < 5，无复杂编排需求                             │
│  □ 单机部署即可满足                                         │
│  □ 团队缺乏 K8s 经验且无时间学习                            │
│  □ 早期 MVP，运维成本敏感                                   │
├─────────────────────────────────────────────────────────────┤
│  轻量替代方案：                                              │
│  · Docker Compose — 单机多容器、开发环境                    │
│  · K3s — 边缘/IoT/CI（轻量 K8s）                            │
│  · Serverless — 事件驱动、无需管基础设施                    │
└─────────────────────────────────────────────────────────────┘
```

**快速判断**：问"我需要跨多台机器自动调度吗？"——不需要则先用 Docker Compose。
