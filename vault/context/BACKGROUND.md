## 2. 协作规范 (Collaboration Norms)

### 2.1 多 Agent 协作黄金法则

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                     💡 多 Agent 协作黄金法则                                     │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│   🏆 「改源不改生，删前先搜用」                                                  │
│                                                                                 │
│   完整版:                                                                        │
│   • 要修改自动生成的文件 → 找到生成器脚本，改源头                                │
│   • 要删除任何代码/方法 → 先搜索是否有调用方                                     │
│                                                                                 │
│   安全前提:                                                                      │
│   ✅ 添加新方法/属性 → 安全，不会破坏现有代码                                    │
│   ⚠️ 删除/重命名接口 → 危险，必须先搜索调用方                                    │
│   ⚠️ 修改生成器脚本 → 危险，影响所有派生文件                                     │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 协作原则

1. **每个 Agent 专注单一职责**，避免职责重叠
2. **通过 Git 协调代码变更**，避免冲突
3. **定期同步进度**，确保各 Agent 工作方向一致
4. **识别「源文件」与「派生文件」**，避免修改冲突
5. **重要决策由人统一做出**，Agent 负责执行

### 2.3 自动化决策

```
十同一异，脚本无疑；五下独特，手动稳妥
```

- 10+ 个相似操作 → 写脚本自动化
- 5 个以下独特操作 → 手动执行更快

## 3. 学习与写作偏好 (Learning & Writing Preferences)

### 3.1 解释偏好: 双车道模式

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                        解释偏好: 双车道模式                                       │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│   Lane 1: 机理优先 (Mechanism-first)                                            │
│   ─────────────────────────────────────────                                     │
│   先讲底层原理、数学模型、因果链条                                               │
│   适合: 技术深度分析、系统设计、调试排查                                         │
│                                                                                 │
│   Lane 2: 类比其次 (Analogy-second)                                             │
│   ─────────────────────────────────────────                                     │
│   用类比帮助理解，但必须标注类比失效条件                                          │
│   适合: 概念入门、跨领域沟通、快速对齐                                           │
│                                                                                 │
│   ⚠️ 类比使用规范:                                                              │
│   • 类比后必须说明「类比失效边界」                                               │
│   • 不能用类比替代机理解释                                                       │
│   • 先机理，后类比（不可反过来）                                                 │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 3.2 KNOW 文档格式偏好

```markdown
# 标题

> Source / Context / Core Insight

---

## Golden Rule (黄金准则)
🏆 一句话总结

---

## The Essence (核心精华)
ASCII diagram 展示核心概念

---

## Deep Dive (深入分析)
详细分析和案例

---

## Action Items (行动清单)
可执行的建议
```

### 3.3 Commit Message 偏好

- 标题和描述用**简体中文**
- emoji 前缀标识类型: 🚀 feat / 🐛 fix / 📚 docs / 🔧 chore / ♻️ refactor
- 详细描述变更内容，用 ASCII diagram 说明结构
- 末尾添加 Claude Code 签名
