**最后更新**: 2025-11-02
**重大更新**: 新增"零容忍代码政策"，明确禁止任何代码和技术术语
**负责人**: Yoach 文档工作组

## 📋 文档说明

本指南用于规范 Yoach 团队在撰写「ORG 文档（Organization-level Module Introduction）」时的标准结构、内容要求和协作流程。

**ORG 文档的核心目的**：
- **对齐理解**：向全团队（产品、设计、开发、教练）快速说明"模块是什么"
- **定义先行**：在编写详细 PRD 之前，先统一模块的核心定义和设计原则
- **高层概念**：聚焦"是什么"和"为什么"，而非"怎么做"的实现细节

**文档流程**：
```
ORG 文档（对齐"模块是什么"）
    ↓
PRD 文档（详细设计"如何实现"）
    ↓
技术实现（开发代码）
```

**核心原则**：
- ✅ **精准简洁** - 聚焦核心定义，避免冗余内容（目标 300 行）
- ✅ **高层对齐** - 面向全团队，统一理解模块定位和价值
- ✅ **自然语言** - 使用描述性语言，**❌ 严禁任何代码示例（Swift/JSON/SQL等）**
- ✅ **用户视角** - 强调用户如何使用模块，而非系统如何实现

> **⚠️ 重要提醒**：ORG 文档是**高层概念文档**，目标读者包括非技术人员（产品经理、教练团队）。
> **绝对禁止**包含任何形式的代码示例（Swift、JSON、SQL、API调用等），所有技术实现细节应放在 PRD 文档中。

**适用场景**：
- 新模块设计完成后，需要向团队介绍模块定位和核心概念
- 现有模块升级到新版本，需要更新定义和设计原则
- 跨团队协作时，需要统一对模块的理解

**文档命名规范**：
```
ORG_[MODULE].md

示例：
- ORG_KSLIDER.md - K-Slider 模块介绍
- ORG_AIPLAN.md - AI训练计划模块介绍
- ORG_YOCHAT.md - YoChat模块介绍
```

## 1. ORG 文档目标与定位

### 1.1 什么是 ORG 文档？

**ORG 文档（Organization-level Module Introduction）**是用于向团队成员快速介绍某个功能模块的**高层定义文档**，重点说明：
- 模块是什么？（核心定义）
- 为什么设计这个模块？（设计原则）
- 用户如何使用？（用户体验流程）
- 模块与其他模块的关系？（系统定位）

### 1.2 ORG 文档 vs PRD 的区别

| 维度 | ORG 文档 | PRD 文档 |
|------|--------------|-------------------|
| **目标读者** | 全团队（产品、设计、开发、教练） | 产品团队、开发团队 |
| **内容深度** | 高层概念，精准简洁 | 详细需求、技术实现 |
| **文档长度** | 200-300 行（精简） | 400+ 行（详尽） |
| **章节数量** | 5 个核心章节 | 7-10 个详细章节 |
| **语言风格** | **纯自然语言，零代码** | 包含代码、数据结构 |
| **聚焦点** | 定义、原则、用户体验 | 需求、实现、验收 |
| **代码示例** | **❌ 绝对禁止** | ✅ 允许且鼓励 |

### 1.3 ORG 文档的核心价值

1. **快速对齐**: 新成员 10 分钟内理解模块核心
2. **统一语言**: 全团队使用统一的模块定义和术语
3. **高层指引**: 为 PRD 提供指导原则，确保 PRD 从正确的模块定义出发
4. **用户视角**: 强调用户如何使用，而非系统如何实现

### 1.4 适用场景

**必须编写 ORG 文档的情况**：
- ✅ 新模块设计完成，需要向团队介绍
- ✅ 模块升级到新版本，定义发生重大变化
- ✅ 跨团队协作项目，需要统一模块理解

**不需要编写 ORG 文档的情况**：
- ❌ 小功能点或子功能（应归入上层模块 ORG）
- ❌ 纯技术实现细节（应写入技术文档）
- ❌ 临时性实验功能（可用 RFC 或 Tech Spec）

## 2. ORG 文档结构规范

### 2.1 文档头部元信息

**必填字段**：
```markdown
# [模块名称]

**版本**: vX.Y.Z
**文档类型**: ORG 文档（Organization-level Module Introduction）
**创建日期**: YYYY-MM-DD
**目标读者**: 产品团队、开发团队、教练团队

**文档目的**: 本文档用于对齐全团队对 [模块名称] 的理解，
定义"[模块名称] 是什么"，为后续编写详细 PRD 文档提供基础。
```

**版本号规则**：
- **vX.0.0**: 模块首次定义
- **vX.Y.0**: 核心定义或设计原则重大变化
- **vX.Y.Z**: 小幅更新或补充说明

### 2.2 必备章节结构（5 个核心章节）

ORG 文档**必须包含且仅包含**以下 5 个核心章节：

```markdown
## 📑 目录

- [1. 模块定义](#1-模块定义)
- [2. 核心设计原则](#2-核心设计原则)
- [3. 模块结构与组成](#3-模块结构与组成)
- [4. 用户体验流程](#4-用户体验流程)
- [5. 模块定位与关系](#5-模块定位与关系)
```

**⚠️ 严格禁止代码示例**：
- ❌ **绝对禁止任何代码** - Swift、SwiftUI、JSON、SQL、API调用等
- ❌ **绝对禁止技术术语** - 变量名、函数名、类名、协议名等
- ❌ **绝对禁止数据结构** - 数据模型、接口定义、数据库schema等
- ✅ **仅使用自然语言** - 描述性语言说明"用户看到什么"和"用户做什么"

**内容要求**：
- ✅ **保持自然语言** - 使用描述性语言说明模块功能和用户体验
- ✅ **高层对齐** - 帮助全团队快速理解"模块是什么"，为后续 PRD 打好基础
- ✅ **用户视角** - 第4章专门描述用户如何使用模块
- ✅ **所有技术细节留给 PRD** - ORG 文档只说"是什么"，PRD 文档说"怎么做"

### 2.3 子章节数量限制

**每个核心章节最多 5 个子章节**

**示例**：
```markdown
## 2. 核心设计原则

### 2.1 原则 #1: 单一焦点，精准反馈
### 2.2 原则 #2: 实时性至上，零感知延迟
### 2.3 原则 #3: 蓝紫渐变，科技精准表达
### 2.4 原则 #4: 浮动布局，视觉焦点平衡
### 2.5 原则 #5: 渐进激励，心理正向强化
```

**超过 5 个子章节时的处理方法**：
- 合并相似子章节
- 将部分内容移至相关 PRD 文档
- 使用表格或列表代替独立章节

### 2.4 文档尾部信息

**必填字段**：
```markdown
---

**文档维护**: Yoach 产品团队
**最后更新**: YYYY-MM-DD
```

**可选字段**：
```markdown
## 📂 相关文档

- 📘 [技术文档](./技术文档.md)
- 📙 [PRD_XX01](./PRD_XX01.md)
```

## 3. 内容编写要求

### 3.1 核心章节内容指南

#### 章节 1: 模块定义

**目标**: 用一句话定义模块，说明核心特征和系统定位

**必备内容**：
1. **一句话定义**（高度概括，30-50 字）
2. **核心特征**（3-5 个特征，使用表格）
3. **系统定位**（模块在整体系统中的角色）

**示例**：
```markdown
### 1.1 一句话定义
> K-Slider 是 YoachApp AR 训练系统的单一关键动作要领实时反馈组件

### 1.2 核心特征
| 特征 | 说明 | 独特价值 |
|------|------|----------|
| 单一焦点评价 | 仅聚焦一个关键动作要领 | 避免信息过载 |

### 1.3 系统定位
[ASCII 图示展示模块在系统中的位置]
```

---

#### 章节 2: 核心设计原则

**目标**: 说明模块设计遵循的核心原则（2-5 个）

**必备内容**：
1. **原则清单**（2-5 个原则）
2. **每个原则的说明**：核心理念、具体体现、设计要求

**示例**：
```markdown
### 2.1 原则 #1: 单一焦点，精准反馈

**核心理念**: 人类注意力资源有限，K-Slider 通过只显示一个关键指标，
帮助用户集中精力突破最重要的技术瓶颈。

**具体要求**:
- 每个训练阶段只绑定一个明确的关键点
- 评分算法简单透明：偏差越小 → 得分越高
- 数据源必须是可量化、单维度、明确定义的指标
```

**原则数量建议**：
- 2-3 个原则：适合简单模块
- 3-5 个原则：适合复杂模块
- 不建议超过 5 个原则

---

#### 章节 3: 模块结构与组成

**目标**: 说明模块的结构、组成部分及其关系

**必备内容**：
1. **结构图示**（ASCII 图）
2. **组成部分说明**（3-5 个核心组件）
3. **关系描述**（组件间如何协同）

**示例**：
```markdown
### 3.1 整体结构

K-Slider 组件（34×226 pt）
├── 视觉层：轨道背景、进度填充、刻度点
├── 数据绑定层：进度值输入、响应式更新
├── 布局层：固定尺寸、右侧中部垂直居中
└── 动画层：平滑过渡、节流机制

### 3.2 核心组件

**组件 #1: 轨道背景**
- 功能：提供进度条的视觉容器和边界定义
- 视觉：圆角矩形，半透明黑色，34×226 pt
```

**❌ 严格禁止**：
- 不得包含任何代码示例（Swift、JSON等）
- 不得使用技术类名（VStack、@Published等）
- 仅使用自然语言描述"视觉呈现"和"功能作用"

---

#### 章节 4: 用户体验流程

**目标**: 描述用户如何使用模块，展示完整的用户旅程

**必备内容**：
1. **核心用户旅程**（从进入到完成的完整流程）
2. **典型使用场景**（2-3 个真实场景示例）

**示例**：
```markdown
### 4.1 核心用户旅程

【场景】用户进行网球正手挥拍训练，重点纠正"手腕下压角度"问题

步骤 1: 进入 AR 训练视图
    → 右侧出现 K-Slider，初始显示 0%
    → 系统提示:"本次重点：手腕下压 30° 击球"

步骤 2: 开始模仿动作
    → 用户平腕击球，K-Slider 实时检测显示 33%
    → 用户意识到手腕角度不对

步骤 3: 调整动作，观察反馈
    → 用户尝试下压手腕，K-Slider 立即上升：33% → 67% → 85%
    → 用户获得正向激励："有效果！"

### 4.2 典型使用场景

**场景 1: 网球正手挥拍 - 手腕下压角度纠正**

**用户背景**：中级玩家，正手击球时手腕角度不足

**用户体验**：
1. 第一次尝试，习惯性平腕击球（0° 下压）→ K-Slider 显示 33%
2. 尝试下压手腕到 15° → K-Slider 上升到 67%
3. 继续调整到 28° → K-Slider 达到 96%，手机震动
4. 经过练习，稳定在 29-31° 区间 → 动作定型

**关键价值**：将"模糊的感觉"转化为明确的量化反馈
```

**注意事项**：
- 使用自然语言描述用户行为
- 用流程图或步骤列表展示旅程
- 聚焦用户视角，说明"用户做什么"而非"系统如何实现"

---

#### 章节 5: 模块定位与关系

**目标**: 说明模块与其他模块的关系和差异

**必备内容**：
1. **对比表格**（本模块 vs 其他相关模块）
2. **协同方式**（如果有模块间协作）
3. **核心价值总结**

**示例**：
```markdown
### 5.1 与其他训练反馈模块的区别

| 维度 | K-Slider | AR 骨架叠加 | 传统评分 | 音效反馈 |
|------|----------|-------------|----------|---------|
| **评价维度** | 单一关键要领 | 无评分 | 单一总分 | 单一事件 |
| **反馈时机** | 实时（30 FPS） | 实时显示 | 事后 | 事后 |
| **用户关注度** | 高（主动关注） | 低（被动查看） | 高（结果导向） | 低（辅助） |

### 5.2 协同方式

**与 MediaPipe 姿态检测的协同**：
MediaPipe 输出 33 个关键点 → K-Slider 提取关键点 → 计算角度/距离
→ 对比理想值 → 转换为 0-100% 进度值 → 实时更新视觉

### 5.3 核心价值

**对用户的价值**：
1. 精准定位问题 - 明确指出"肘部位置准确度只有 40%"
2. 即时学习闭环 - 调整动作 → 立即反馈 → 继续优化
3. 量化进步可视化 - 追踪特定动作的进步曲线

**对产品的价值**：
1. 差异化竞争优势 - 市场上少有"单一关键要领实时反馈"产品
2. 用户留存提升 - 实时反馈提升体验满意度
3. 数据资产沉淀 - 积累用户在不同技术动作上的掌握度数据
```

---

#### 原则 #3: 自然语言优先

**要求**：
- ✅ 使用描述性语言说明功能
- ✅ 用"视觉呈现"代替"技术实现"
- ✅ 用"数据流向"代替"代码逻辑"
- ❌ **绝对禁止 Swift 代码、JSON 结构、API 定义等任何技术细节**

**正确 vs 错误对比**：

| ❌ 错误写法（包含代码） | ✅ 正确写法（纯自然语言） |
|------------------------|--------------------------|
| `VStack { Spacer(); Rectangle().fill(...) }` | 底部对齐填充，从下往上生长 |
| `@Published var progress: Double` | 进度值实时更新显示 |
| `struct RizDevice { id: Int, color: RGB }` | Riz 设备包含编号和颜色信息 |
| `GET /api/training/history` | 获取用户历史训练数据 |
| `CoreBluetooth scanning...` | 扫描附近的蓝牙硬件设备 |

**核心规则**：
> 如果你写的内容需要技术背景才能理解，说明你违反了 ORG 文档规范。
> ORG 文档应该让产品经理和教练团队也能轻松理解。

## 4. 审核与维护流程

### 4.1 文档创建流程

```
步骤 1: 确定模块范围
  ↓
步骤 2: 创建文档草稿（使用 ORG_[MODULE].md 命名）
  ↓
步骤 3: 填写 5 个核心章节（每章节不超过 5 个子章节）
  ↓
步骤 4: 团队评审（产品、开发、教练团队检查）
  ↓
步骤 5: 发布文档（更新状态为 "Approved"）
  ↓
步骤 6: 基于 ORG 文档编写详细 PRD 文档
```

### 4.2 审核清单

**内容完整性检查**：
```
✔ 文档包含 5 个核心章节
✔ 每个章节不超过 5 个子章节
✔ 一句话定义清晰准确
✔ 核心设计原则明确（2-5 个）
✔ 用户体验流程完整
✔ 模块定位与关系清晰
```

**格式规范检查**：
```
✔ 文档命名符合 ORG_[MODULE].md 规范
✔ 文档长度控制在 200-300 行
✔ ❌ 绝对零代码：无 Swift、JSON、SQL、API等任何代码
✔ ❌ 绝对零技术术语：无类名、函数名、协议名等
✔ 表格格式正确，列数不超过 5 列
✔ 无四级以下标题
```

**代码检查清单（必须全部为否）**：
```
❌ 是否包含代码块（```swift、```json等）？
❌ 是否包含技术类名（VStack、ObservableObject等）？
❌ 是否包含数据结构定义（struct、class等）？
❌ 是否包含API调用示例（GET、POST等）？
❌ 是否包含技术框架名称（CoreBluetooth、SwiftUI等）？

✅ 以上全部为"否"，则通过代码检查
```

**质量标准检查**：
```
✔ 内容精准简洁，无冗余描述
✔ 使用团队统一术语
✔ 聚焦高层概念和用户体验
✔ 版本号和日期正确
✔ 相关文档链接有效
```

### 4.3 更新与维护

**触发更新的场景**：
1. **版本升级**：模块核心定义变化
2. **设计原则调整**：新增或修改核心设计原则
