## 执行摘要

### 反直觉洞见

> **"程序员不学 CAD 不是因为懒，而是因为 CAD 的交互范式与编程完全相反——鼠标点击 vs 文本代码、二进制文件 vs 版本控制、GUI 依赖 vs 命令行自动化。"**

这个洞见解释了为什么即使有 CadQuery、OpenSCAD 这些"代码化 CAD"工具，大多数程序员仍然选择用 3D 打印社区下载的 STL 文件，而不是自己设计——**现有工具仍然需要学习 CAD 概念**（布尔运算、挤出、圆角），只是换了一种语法表达。

### Why This Matters

CAD 市场正在经历 AI 革命。61% 的用户寻求 AI 辅助设计，33% 的 AI 集成增长率说明这是最热门的方向。但当前的 AI CAD 工具（如 Zoo.dev）仍然是"为设计师添加 AI"，而不是"为程序员设计工具"。

| 用户类型 | 现有解决方案 | 痛点 |
|----------|-------------|------|
| **机械工程师** | SolidWorks, Fusion 360 | 工具成熟，无痛点 |
| **设计师学 CAD** | Zoo.dev, Onshape | 学习曲线降低，但仍需学习 |
| **程序员做硬件** | CadQuery, OpenSCAD | 需要学习 CAD 概念 + 新语法 |
| **程序员不想学** | **???** | **市场空白** |

**MechCode 的机会**：填补"程序员根本不想学 CAD 概念"这个市场空白。用自然语言描述需求，用 YAML 文件管理参数，用 Python API 集成到自动化流程。

### 核心结论

1. **Zoo.dev 的局限**：Text-to-CAD 领先者，但用 KCL（自定义语言），程序员仍需学习新语法
2. **CadQuery 的局限**：Python 生态但无 AI，仍需理解 OCCT 内核概念
3. **MechCode 的破局路径**：自然语言 + YAML + 图片逆向 = "零 CAD 概念"的设计体验

## 1.3 AI 3D 生态系统：上游与互补

> **关键洞见**：Image-to-3D / Video-to-3D AI 工具（Tripo, Meshy, Rodin, Luma AI）是 MechCode 的**上游/互补**，而非竞争对手。核心区别在于**输出格式**。

> **深度分析目的**：理解主要竞品的技术路线、优势和局限，为 MechCode 定位提供依据。

## 2.1 Zoo.dev (KittyCAD) - 首要威胁

**定位**: Text-to-CAD 领导者 | **融资**: $10.12M (Seed) | **背景**: Sequoia、GitHub 联合创始人投资

**深度分析**：Zoo.dev（原 KittyCAD）是目前 Text-to-CAD 领域最有资金支持的玩家。由前硬件工程师创立，2021 年成立，2024 年 1 月更名为 Zoo。其核心产品是基于 AI 的 Text-to-CAD API 和自研的 KCL（KittyCAD Language）参数化语言。

**关键技术**：
- **Text-to-CAD API (ML-ephant)**：自然语言转 CAD 模型
- **KCL 语言**：专为 CAD 设计的领域特定语言
- **B-Rep 输出**：直接生成边界表示法（非点云/网格）
- **STEP 兼容**：可导入任何 CAD 软件编辑

**商业模式**：
- 免费：40 分钟/月
- 付费：$0.50/分钟（约 $30/小时）
- 企业：定制定价

**Zoo 声称**：
> "内部测试显示，我们的生成式 AI 可以为设计某些复杂零件节省高达 95% 的时间。"

## 2.2 CadQuery - 开源标杆

## 2.3 OpenSCAD - CSG 先驱

**定位**: 开源 CSG 脚本 CAD | **历史**: 2010+ | **背景**: 3D 打印社区标准

**深度分析**：OpenSCAD 是代码化 CAD 的先驱，在 3D 打印爱好者社区广泛使用。其核心理念是"程序员的 CAD"——用代码而非鼠标设计。但它基于 CSG（构造实体几何）范式，有固有的技术限制。

**技术限制**：
- **无法输出 STEP**：只能输出 STL（网格），不能输出真曲线
- **CSG vs BREP**：布尔运算效率低，复杂模型容易出错
- **自定义语法**：不是标准编程语言，无法复用 Python 生态

```

│                                                                         │

│  ─────────────────────────────────────────────────────────────────────  │
│  原理: 通过布尔运算（并、交、差）组合基本形状                            │
│                                                                         │
│    ◯ + ◻ = ◯◻    (并集)                                                │

│                                                                         │

### MechCode .mech 文件示例

```yaml
# motor_mount.mech - 由 MechCode AI 自动生成
# 可在任意文本编辑器中编辑，支持 git diff

version: "1.0"
part_type: motor_mount
description: "NEMA 17 stepper motor mounting bracket"

# 参数化变量 - 修改这里自动更新模型
parameters:
  motor_type: nema_17      # AI 自动识别标准规格
  thickness: 5             # mm
  material: aluminum_6061  # 用于质量计算

# 自动派生的尺寸 (由 motor_type 决定)
derived:
  faceplate_size: 42.3     # NEMA 17 标准
  mounting_hole_spacing: 31 # NEMA 17 标准
  shaft_hole_diameter: 22  # 中心孔
  mounting_hole_diameter: 3.2  # M3 通孔

# 可选特征
features:
  mounting_holes: true
  shaft_hole: true
  chamfer_edges: 0.5       # mm

# 输出配置
output:
  formats: [step, stl]
  units: mm
```

**关键差异**：
- **YAML 格式**：程序员熟悉，任意编辑器可编辑
- **参数化**：修改 `thickness: 5` → `thickness: 8`，模型自动更新
- **标准件知识**：输入 `nema_17`，AI 知道所有相关尺寸
- **Git 友好**：文本文件，完美支持 diff/merge/blame

## 可迁移规则：程序化 CAD 产品的 5 条普适规律

> **本章价值**：以下规则不仅适用于机械设计，也适用于任何试图用代码/AI 替代传统 GUI 工具的产品。

### 规则 #1: 隐藏复杂性，而非教育用户

**模式**：
```
❌ 错误模式: 给用户一个"更简单的语法"学习 CAD 概念
✅ 正确模式: 用 AI 完全隐藏 CAD 概念，用户只需描述需求
```

**为什么有效**：用户不是想"用更简单的方式学 CAD"，他们是"根本不想学 CAD"。CadQuery 和 OpenSCAD 的问题是仍然需要学习挤出、布尔运算等概念。

**如何应用**：设计产品时问自己——"用户需要知道多少领域知识才能使用？"如果答案超过零，考虑用 AI 抽象掉。

### 规则 #2: 文件格式是护城河

**模式**：
```
二进制格式     →  厂商锁定，但无法版本控制
通用文本格式   →  可移植，但无差异化
专有文本格式   →  差异化 + 可版本控制 + 生态锁定
```

**为什么有效**：.mech 的价值不仅是"人类可读"，而是"成为程序员之间交换参数化设计的标准格式"。一旦社区开始围绕格式构建工具，就形成了网络效应。

**如何应用**：如果你的产品有核心数据结构，考虑定义开放的文本格式规范，鼓励社区构建生态。

### 规则 #3: CLI 优先，GUI 可选

**模式**：
```
设计师工具: GUI 优先，CLI 后补
程序员工具: CLI 优先，GUI 可选
```

**为什么有效**：程序员的工作流是终端 + 文本编辑器 + Git。GUI 在这个流程中是"额外的窗口"。Zoo.dev 的 Web Studio 对设计师友好，但对程序员是负担。

**如何应用**：所有功能都应该可以通过 CLI 完成。GUI 只是 CLI 的可视化封装，而非独立入口。

### 规则 #4: 双向能力创造差异化

**模式**：
```
单向工具: 输入 A → 输出 B (所有竞品)
双向工具: 输入 A → 输出 B  AND  输入 B → 输出 A (独有)
```

**为什么有效**：逆向工程（照片→模型）是现有工具的空白。这不仅是功能差异，更是使用场景的扩展——覆盖"复制损坏零件""数字化老旧设备"等场景。

**如何应用**：思考你的产品能否支持"反向操作"。图片→代码、语音→文本、视频→脚本等，双向能力往往能覆盖被忽视的使用场景。

### 规则 #5: 标准件知识是 AI 的真正价值

**模式**：
```
❌ 错误的 AI 价值: "用 AI 画出用户描述的几何形状"
✅ 正确的 AI 价值: "AI 知道 NEMA 17 电机的所有标准尺寸"
```

**为什么有效**：如果用户需要说"42.3mm × 42.3mm，安装孔间距 31mm"，AI 只是一个语法翻译器。真正的价值是用户说"NEMA 17"，AI 就知道所有相关尺寸——这才是"领域知识"的价值。

**如何应用**：构建领域知识库，让 AI 成为"懂行的专家"，而非"听话的翻译器"。

## Key Takeaways：多角色行动建议

### For Solo Makers/Hobbyists (个人创客/爱好者)

1. **不要学 SolidWorks**：如果你只是需要简单的机械零件，学习 CAD 的 ROI 太低
2. **用描述替代设计**：尝试用自然语言描述你需要的零件，而不是思考"怎么画出来"
3. **拍照复制零件**：坏掉的零件？拍张照片，让 AI 逆向工程，比从头设计快 10 倍

### For Robotics Engineers (机器人工程师)

1. **Python API 集成**：将机械零件生成集成到你的机器人代码库
2. **参数化设计**：用变量定义尺寸，方便快速迭代
3. **Git 管理机械设计**：.mech 文件可以和代码一起版本控制，机电一体化管理

### For Hardware Startup Founders (硬件创业者)

1. **团队技能复用**：如果团队主要是软件工程师，不需要专门招机械工程师做简单原型
2. **快速迭代**：从想法到 3D 打印验证可以在一天内完成，而不是等待外包
3. **成本控制**：省下 SolidWorks 许可费（$4000/人/年 × 团队人数）

## 术语表 (Glossary)

| 英文术语 | 中文翻译 | 定义 |
|----------|----------|------|
| **CAD** | 计算机辅助设计 | Computer-Aided Design，用计算机进行设计工作 |
| **BREP** | 边界表示法 | Boundary Representation，用数学曲面定义实体边界 |
| **CSG** | 构造实体几何 | Constructive Solid Geometry，用布尔运算组合基本形状 |
| **OCCT** | Open CASCADE 技术 | CadQuery 和 MechCode 使用的开源 CAD 内核 |
| **STEP** | 标准产品模型交换 | ISO 标准，工业级 CAD 交换格式 |
| **STL** | 立体光刻 | 3D 打印常用格式，仅网格无参数化 |
| **NEMA** | 美国电气制造商协会 | 定义电机尺寸标准（如 NEMA 17, NEMA 23） |
| **KCL** | KittyCAD Language | Zoo.dev 的自定义 CAD 语言 |
| **.mech** | MechCode 文件 | MechCode 的 YAML 参数化设计文件 |
| **Parametric** | 参数化 | 用变量定义尺寸，修改变量自动更新模型 |
| **Reverse Engineering** | 逆向工程 | 从实物/照片还原设计的过程 |
| **CLI** | 命令行界面 | 通过终端命令操作的程序界面 |
| **Mesh** | 网格 | 由三角形/四边形面片组成的 3D 模型，用于游戏/视觉，无法精确制造 |
| **NeRF** | 神经辐射场 | Neural Radiance Fields，用神经网络表示 3D 场景的技术 |
| **Quad Mesh** | 四边形网格 | 由四边形面片组成的网格，拓扑更规整，便于动画和编辑 |
| **GLB/GLTF** | GL 传输格式 | 用于 Web 和 AR/VR 的 3D 模型格式 |
| **FBX** | Filmbox | Autodesk 的 3D 交换格式，广泛用于游戏和动画 |
| **OBJ** | Wavefront Object | 通用 3D 模型格式，仅含几何信息 |
